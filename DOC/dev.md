# 机械设计工具箱开发文档（Markdown 版）

## 1. 项目概述

### 1.1 项目名称

**Mechanical Toolbox / 机械设计工具箱**

### 1.2 项目定位

这是一个面向机械设计、工艺、选型与校核场景的桌面端工程工具箱。
它不是单一“公式计算器”，而是一个围绕 **计算、选型、校核、查询、留痕、导出** 建立的工程辅助平台。

首批目标模块以高频、标准化、重复劳动重、容易出错的场景为主，包括：

* 密封设计
* 轴承选型与寿命估算
* 传动计算
* 公差与配合
* 材料属性与选材
* 紧固件与标准件速查
* 常用单位换算与工程辅助计算

### 1.3 立项背景

机械设计行业里，很多日常工作仍依赖 Excel、纸质手册、PDF 标准、经验公式和分散网页工具。
以密封设计为例，公开标准和厂商资料本身就是按 **尺寸、沟槽、材料、工况、安装、失效** 多维组织的，而不是单纯给一个计算公式；SKF 的轴封资料也明确把选型建立在温度、速度、压差、润滑、偏心和安装条件等因素上。([SKF][1])

## 2. 项目目标

## 2.1 总目标

构建一个适合机械工程师在桌面环境中使用的专业工具箱，解决以下问题：

* 降低手工查表和重复计算成本
* 降低单位换算、参数遗漏、公式误用带来的错误
* 把标准、经验和计算流程沉淀为可复用的工具
* 让结果可保存、可导出、可回溯
* 为后续扩展企业知识库、项目模板、案例库预留基础

## 2.2 产品目标

第一阶段不追求“大而全”，而是先形成一个可用、可演示、可扩展的桌面产品：

* 支持多模块并行扩展
* 支持标准库与材料库管理
* 支持工程计算结果导出
* 支持本地离线优先
* 支持后续插件化

## 6. 核心模块需求

## 6.1 密封模块

### 6.1.1 O 形圈计算页面

这是你发图里类似的页面，但在整个产品里只是一个子页面，不是全部产品。

#### 输入项

* 密封类型：静密封 / 动密封
* 安装方式：径向内密封 / 径向外密封 / 轴向密封
* 标准体系：ISO / GB / AS568 / 自定义
* 孔径、轴径、沟槽尺寸
* O 形圈内径、线径
* 公差与偏差
* 压力、温度、介质
* 材料、硬度
* 是否挡圈
* 单位制

#### 输出项

* 推荐规格
* 推荐沟槽尺寸
* 压缩率
* 拉伸率
* 填充率
* 间隙与挤出风险
* 材料适用性提示
* 安装提示
* 风险预警

#### 交互要求

* 左侧示意图与字段联动
* 字段修改后实时刷新结果
* 标准匹配结果可一键带入
* 超限项高亮
* 结果支持快照对比

### 6.1.2 失效诊断

* 输入现象：泄漏、磨损、切边、挤出、硬化、龟裂等
* 输出可能原因与建议措施
* 可关联到当前计算结果

### 6.1.3 标准与依据

国家标准公开检索系统可以查到 O 形圈相关标准体系，例如沟槽尺寸、公差、材料等分层标准；这正适合做成“规则 + 数据库 + 校核”的产品结构。([Electron][2])

## 9. 技术方案

## 9.1 技术选型

建议采用：

* **Electron**：桌面容器
* **React + TypeScript**：前端界面
* **Vite**：构建
* **SQLite**：本地数据存储
* **Zustand / Redux Toolkit**：状态管理
* **React Router**：页面路由
* **ECharts / Chart.js**：图表展示
* **electron-builder 或 Electron Forge**：打包发布

Electron 本身采用 Chromium 的多进程架构，主进程负责应用生命周期和窗口管理，渲染进程负责界面；预加载脚本可作为受控桥接层暴露桌面能力。这个架构很适合“计算引擎、文件系统、打印/导出、界面”分层。([Electron][2])

## 9.2 推荐架构

```text
┌─────────────────────────────┐
│         Electron Main       │
│ 窗口管理 / 文件系统 / 导出 / 更新 │
└──────────────┬──────────────┘
               │ IPC
┌──────────────▼──────────────┐
│          Preload API        │
│ 安全桥接：project/report/db  │
└──────────────┬──────────────┘
               │ contextBridge
┌──────────────▼──────────────┐
│     Renderer (React UI)     │
│ 页面 / 表单 / 结果 / 图示 / 状态 │
└──────────────┬──────────────┘
               │
┌──────────────▼──────────────┐
│  Domain Layer / Calc Engine │
│ 密封 / 轴承 / 传动 / 公差规则  │
└──────────────┬──────────────┘
               │
┌──────────────▼──────────────┐
│ SQLite / JSON / Rule Packs  │
│ 标准库 / 材料库 / 项目数据     │
└─────────────────────────────┘
```

## 9.3 为什么这样拆

* UI 层负责交互和展示
* 计算层负责公式、校核、推荐逻辑
* 数据层负责标准表、材料表、项目文件
* 主进程负责文件、打印、更新、系统集成

这样做的好处是：后面新增轴承模块或齿轮模块时，不需要重写整套 UI 框架。

## 10. 安全与 Electron 实施要求

Electron 官方安全文档明确建议不要为远程内容启用 Node.js 集成，并建议启用 `contextIsolation`；而且 `contextIsolation` 自 Electron 12 起默认启用。对于你的产品，最佳做法就是 **本地页面 + preload 白名单 API + 最小权限 IPC**。([Electron][3])

### 10.1 安全基线

* `nodeIntegration: false`
* `contextIsolation: true`
* `sandbox: true`（能开则开）
* preload 仅暴露有限 API
* 不在 renderer 中直接操作文件系统
* 不加载不受控远程页面
* 导出、更新、打开文件均走主进程

### 10.2 自动更新

Electron 官方 `autoUpdater` 支持 macOS 和 Windows，Linux 没有内建支持；同时 macOS 自动更新要求应用签名。你的产品如果以后做商用分发，更新体系从第一版就要预留。([Electron][4])

## 12. 计算引擎设计

## 12.1 原则

* 公式和规则分离
* 输入校验先于计算
* 结果必须带出处/规则版本
* 支持“推荐”和“校核”两种模式

## 12.2 引擎层建议拆分

```text
/domain
  /oring
    input.ts
    formulas.ts
    validator.ts
    recommender.ts
    report.ts
  /bearing
  /transmission
  /fit
```

## 12.3 输出标准

每个模块输出统一结构：

```ts
type CalcResult<TInput, TOutput> = {
  module: string
  input: TInput
  output: TOutput
  warnings: string[]
  suggestions: string[]
  passItems: string[]
  failItems: string[]
  ruleVersion: string
}
```

## 13.1 通用原则

## 17. 风险与注意事项

## 17.1 产品风险

* 一开始模块铺太大，开发失控
* 规则来源不统一，结果不可信
* 页面很多，但真正可用的深度不够

## 17.2 技术风险

* 计算逻辑散落在组件里，后续难维护
* 主进程和渲染进程职责不清
* 导出、打印、文件读写后期返工

## 17.3 建议

* 先把 O 形圈模块做成“标杆模块”
* 以同一套架构复制到其他模块
* 先做 3 个强模块，不要一次铺 20 个弱模块

## 19. 结论

这个项目最合适的产品形态，不是“一个密封圈页面”，而是：

**一个以工程规则为核心、以桌面端为载体、以模块化扩展为能力边界的机械设计工具箱。**

其中：

* **O 形圈计算页** 是第一批核心页面
* **材料库、标准库、报告中心、项目中心** 是支撑系统
* **轴承、公差、传动** 是后续最自然的扩展路径
* **Electron + React + TypeScript + SQLite** 是比较稳的落地方案，既适合离线工程工具，也符合 Electron 官方推荐的主进程 / 渲染进程 / preload 分层方式与安全实践。([Electron][2])

下一步最值得做的是把这份文档继续细化成 **PRD + 技术设计 + 页面清单 + 数据表结构 + 开发任务拆分**。
