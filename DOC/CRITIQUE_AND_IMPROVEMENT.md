# MechBox (MechCalc) 架构演进与下一代改进计划 (纯本地化与重火力演进版)

> **文档定位**：本文档基于 MechBox 当前的实际代码实现（Vue 3 + Electron + SQLite + Solver 引擎），从资深工业软件架构师的视角，总结已落地的架构重构成果，并针对“100% 纯本地化、超大容量知识库、极致丰富功能”的重度辅助设计软件目标，提出下一阶段（v2.0+）的深度缺陷剖析与演进方案，**并提供具体的底层技术实现路径**。

---

## 1. 核心计算引擎：从“单点求解”向“本地系统级耦合”演进

### 1.1 已落地成果回顾
当前系统已成功在 `src/renderer/engine/solver.ts` 中引入了 **约束求解层 (Solver Layer)**。包含牛顿-拉夫逊法（Newton-Raphson）、二分法及基于惩罚函数的离散优化器。这彻底改变了早期的纯函数“单向映射”，实现了带多约束条件的反向寻优（如轴承选型、螺栓选型等逆向求解）。

### 1.2 当前架构缺陷：缺乏系统级耦合与高级概率分析
**痛点**：
*   **孤岛计算**：目前的求解器仍局限于“单零件”。在实际工业设计中，轴的受力弯曲挠度会直接影响齿轮的载荷分布系数，同时改变轴承的受力分布。这些物理量是强耦合的。
*   **单维度的公差分析**：现有的公差分析通常基于极值法（Worst-case），缺乏基于统计学的批量生产概率预估。

### 1.3 下一代演进方案（v2.0）
1.  **构建本地装配体系统级求解器**：建立跨模块的系统级上下文（System Context）。支持“轴-齿轮-轴承”的传动系统级建模，引入传递矩阵法或轻量级 1D-FEM 在本地求解轴的变形挠度，并自动将结果反哺给轴承和齿轮模块。
    *   **💡 技术实现路径**：在 Rust 核心中构建基于传递矩阵法（TMM）的 1D-FEM 求解器。利用 `nalgebra` 或 `ndarray` 数学库进行稀疏矩阵运算。将轴抽象为多段梁单元（Beam Elements），在节点上施加齿轮啮合力与轴承支承刚度矩阵，联立方程组求解出各节点的变形挠度和转角。
2.  **本地超高频 Monte Carlo (蒙特卡洛) 模拟**：在纯本地环境中，进行数百万次的概率统计模型抽样算法，准确预测批量生产的装配良品率，从而放宽非关键公差，降低制造成本。这完全不依赖云计算，极致榨取本地 CPU 性能。
    *   **💡 技术实现路径**：利用 Rust 的 `rand` 和 `statrs` 库生成正态/偏态分布的随机公差样本。将核心抽样算法编译为 WebAssembly (WASM)，在 Vue 渲染进程中通过 `Web Worker` 开启多线程并行计算。利用 `SharedArrayBuffer` 在主线程与 Worker 间零拷贝传递结果，实现百万次抽样在毫秒级完成，并在 UI 侧用 ECharts 渲染出优美的正态分布概率钟形曲线。

## 2. 数据中心：打造“超大规模本地工程知识库” (100% 离线)

### 2.1 已落地成果回顾
已成功接入 `better-sqlite3`（见 `src/main/db/database.ts`）。实现了本地标准库的按需 SQL 查询，摒弃了低效的 JSON 全量常驻内存模式，为容纳海量数据打下了地基。

### 2.2 当前架构缺陷：数据量较少与深度不足
**痛点**：
*   **资料库单薄**：目前内置的标准件（如轴承型号、螺栓等级）规格覆盖面不足，缺乏美标（ASME）、德标（DIN）、日标（JIS）的全面收录。
*   **高级材料与选型数据匮乏**：缺乏高级疲劳寿命分析所需的 S-N 疲劳曲线数据点、应力-应变非线性材料属性等深度工程资料。

### 2.3 下一代演进方案（v2.0）
**坚守“纯本地构建”原则，拒绝依赖外部 API，将 MechBox 打造为断网环境下的百科全书。**
1.  **内嵌全量全球工业标准库**：大幅扩充本地 SQLite 数据库，收录涵盖 GB/ISO/DIN/JIS/ASME 的全量标准件尺寸、公差、物理参数。
    *   **💡 技术实现路径**：在 `better-sqlite3` 基础上全面启用 SQLite 的 FTS5 (Full-Text Search) 扩展模块。为跨标准的模糊检索（如工程师模糊搜索“M10 沉头”）建立高性能倒排索引；为极高频的查询字段建立 B-Tree 复合索引，保证百万条规格记录查询在 10ms 内返回。
2.  **集成“重火力”工程材料数据库**：在本地数据库中构建多维度的材料物性库，涵盖不同温度下的弹性模量衰减曲线、热膨胀系数非线性变化表、常见材料的 S-N 疲劳曲线及化学兼容性矩阵。
3.  **本地知识库扩展与自定义 (Local Extension)**：提供图形化界面，允许工程师将自己公司内部的“非标件手册”、“企业特有材料”一键导入到 SQLite 的 `user_standards` 表中。
    *   **💡 技术实现路径**：在 Electron 主进程引入 `xlsx` 或 `csv-parse` 引擎，提供标准化的本地 Excel 模板导入功能。通过预定义的 JSON Schema 校验器（如 `ajv`）清洗用户脏数据，利用 SQLite 的 JSON1 扩展将不规则的非标附加属性存入同一行的 JSON 动态字段中，既保证查询效率，又兼顾 Schema 灵活性。

## 3. 跨软件协同：纯本地 CAD 原生插件与双向联动

### 3.1 已落地成果回顾
目前能够将计算结果一键导出为 SolidWorks VBA 宏、FreeCAD Python 脚本或 STEP 组装建议说明，初步打通了从计算数据到 3D 建模的第一道壁垒。

### 3.2 当前架构缺陷：单向生成的体验局限
**痛点**：宏代码生成本质上是“一次性单向抛出”。如果工程师随后在 CAD 中微调了尺寸，MechBox 的参数不会自动更新，容易导致“两层皮”。

### 3.3 下一代演进方案（v2.0）
1.  **研发原生 CAD 本地插件 (Add-in)**：为 SolidWorks / Inventor / SolidEdge 等主流软件开发轻量级的原生插件面板，直接安装在用户的电脑上。
2.  **Local API 与 WebSocket 本机通信**：MechBox 桌面端作为**本地服务端**开启 WebSocket 服务。CAD 本地插件与其建立双向通信通道，实现参数双向捕捉与实时驱动。
    *   **💡 技术实现路径**：在 Electron Main 进程中集成 `ws` 模块，启动常驻本地守护进程（监听如 `127.0.0.1:8321`）。对于 SolidWorks 侧，使用 C# 编写 Add-in，利用 `ClientWebSocket` 连接到 MechBox。两者之间采用强类型的 JSON-RPC 协议交换数据。例如发送 `{"action": "update_dimension", "feature": "O_Ring_Groove_1", "params": {"depth": 1.5}}`，在不产生任何外网流量的情况下实现零延迟的双向参数驱动。

## 4. UI/UX 交互：本地 3D 渲染与参数化可视

### 4.1 已落地成果回顾
基于 Vue 3 + Vite + Pinia 构建了具有现代工程软件质感的界面，成功分离了渲染进程与计算引擎。

### 4.2 当前架构缺陷：缺乏直观的空间几何反馈
**痛点**：参数输入过于抽象。组合参数是否会导致结构干涉（如沟槽过深切穿轴壁），往往只能等导入 CAD 后才能发现。

### 4.3 下一代演进方案（v2.0）
1.  **内嵌 WebGL (Three.js / Vue TresJS)**：在本地离线包中集成 3D 渲染引擎。当用户调整螺栓规格或沟槽尺寸时，右侧实时通过本地参数化算法生成并渲染 3D Mesh 模型（数字孪生），实现“所见即所得”。
    *   **💡 技术实现路径**：在 Vue 组件中挂载 `@tresjs/core` 视图。对于标准件（如螺栓、齿轮），**不使用庞大的 glTF 静态模型文件**。而是通过数学算法动态生成几何体（Procedural Generation），例如利用渐开线方程计算点集，并配合 `THREE.ExtrudeGeometry` 动态拉伸生成齿轮的 3D 网格。这既缩小了安装包体积，又实现了无限级的参数化微调展示。
2.  **本地实时干涉检查**：利用 WebGL 和本地空间计算算法进行碰撞检测。在软件界面的 3D 场景中，将过度挤压或根切的受损区域用红色网格高亮警示。
    *   **💡 技术实现路径**：利用 Three.js 的 `Box3` (AABB 轴对齐包围盒) 进行粗略的碰撞筛选，再结合开源库 `three-mesh-bvh` 构建层级包围体（Bounding Volume Hierarchy），进行精确到面片（Triangle）级别的网格交集计算。将发生干涉的面片传入自定义 Shader (GLSL)，渲染为危险的红色闪烁材质。

## 5. 极致丰富功能扩展：工业级的“本地瑞士军刀”

**基于强大的本地计算核心与海量资料，MechBox 需要从“单一计算工具”跃升为涵盖设计全生命周期的综合性辅助平台：**

1.  **多参数扫描与敏感度分析 (DOE - 参数优化)**：
    *   **功能欠缺**：目前只能进行“单点计算”，工程师无法直观地看到某个参数的变化对整体寿命或强度的趋势影响。
    *   **演进方案**：引入参数化扫描功能（Parameter Sweep）。允许用户设定一个区间的变量（例如：转速从 1000 变到 5000 RPM，步长 100），自动在本地通过 Rust 核心进行批处理计算，并在 UI 端使用 ECharts 绘制出 2D/3D 的寿命/载荷/温度敏感度云图，帮助工程师直观找到“最优设计甜点区”。
2.  **本地自动化工程报告与合规生成 (Compliance Reports)**：
    *   **功能欠缺**：虽然目前有导出 PDF 的计划，但距离工业标准的合规审查（如 ISO 9001 设计评审）还有差距。
    *   **演进方案**：内置基于 LaTeX 引擎的本地排版系统。不仅输出最终结果，更要在报告中以极高颜值的数学公式展示所有的**推导步骤**、**使用的引用标准号（如 GB/T 3452.1）**、**安全系数校验结论**。一键生成带有水印、项目编号、版本追溯的专业级计算书，直接用于图纸归档和甲方审核。
3.  **轻量级本地项目与版本管理 (PLM Lite)**：
    *   **功能欠缺**：计算结果算完即走，没有项目生命周期管理，不同的设计迭代方案难以进行横向比对。
    *   **演进方案**：在 SQLite 中构建基于 Git 思想的本地历史追踪系统（Local Versioning）。工程师可以为一个项目建立多个“分支”（例如：方案A-成本优先，方案B-寿命优先），并能一键生成“方案对比雷达图”。支持将装配体中所有的选型零件一键汇总，自动导出带有 MISUMI/SKF 订货号的 **BOM（物料清单）表格**。
4.  **面向制造的公差映射与成本预估 (DFM 辅助)**：
    *   **功能欠缺**：公差只停留在“是否干涉”的几何层面，脱离了车间的实际加工能力。
    *   **演进方案**：建立公差与加工工艺的映射矩阵（Tolerance-to-Process Mapping）。当工程师选择 IT5 级精密公差时，软件不仅做几何校验，还能在本地知识库中匹配出“需采用外圆磨削工艺”，并根据工艺库给出粗略的“相对制造成本惩罚指数”，引导工程师在满足功能的前提下，尽量采用低成本的宽泛公差（DFM - 面向制造的设计）。
5.  **本地失效分析与专家诊断系统 (Expert System)**：
    *   **功能欠缺**：缺少售后和现场支持的逆向排故能力。
    *   **演进方案**：内置基于专家规则树（Expert System）的失效诊断向导。现场工程师输入失效现象（如：齿面点蚀、O型圈龟裂、轴承发热且有异响），系统在本地匹配出可能的原因（润滑不良、超温、过载），并倒推回选型计算模块进行验算复盘。
6.  **多国标准零部件与材料互换速查系统 (Global Standard Cross-Reference)**：
    *   **功能欠缺**：机械工程师在面对进口设备测绘、维保或出口图纸转换时，经常被多国标准（GB、ISO、DIN、JIS、ASME、GOST）的差异折磨。这不仅限于螺纹，还包括了材料牌号、型钢尺寸、花键参数等，查阅并比对这些分散的 PDF 标准极其耗时且极易出错。
    *   **演进方案**：建立一个无所不包的跨标准“特征聚合与互换代换引擎”。
        *   **多维参数互查**：不仅涵盖所有制式螺纹（BSP、NPT、JIC、UNF、公制）与管接头的大径、小径、底孔与牙型查询，还覆盖**花键/键槽尺寸**、**法兰接口**、**型钢型材截面参数**（如 H型钢、工字钢）的多国标准速查。
        *   **逆向识别向导**：提供可视化的“测量反推助手”。工程师只需输入用卡尺和量角器在现场测得的几个外形数据（如螺纹外径+几扣牙长，或者花键的外径+齿数），系统通过模糊算法自动反推并列出最可能的几项标准规格。
        *   **跨国材料与件代换指南**：当国内难以采购到原装美标材料（如 ASTM A36）或英制轴承时，系统能在本地数据库匹配并推荐**化学成分与力学性能最接近的国标代换材料（如 Q235B）**或公制近似零部件，并标明两者的极限承载差异，大幅度减轻跨国维保与供应链采购的压力。

## 6. 跨平台与底层：Electron 架构下的 Rust 性能榨取

### 6.1 当前状态与局限
目前采用 `electron-vite` 搭配纯 TypeScript 构建，开发效率极高，但在处理海量本地数据检索和复杂的本地系统级耦合运算时，Node.js (V8) 的单线程与计算性能瓶颈会显现。

### 6.2 架构演进（v3.0 预研）：Rust 核心 + Electron/Vue UI
放弃推翻现有生态去完全重写 Tauri，而是在**保持当前 Electron + Vue 3 极佳工程生态的前提下**，将繁重的计算任务平滑下沉到 Rust。
1.  **Rust 核心下沉 (通过 napi-rs 或 WebAssembly)**：
    *   **高密计算重构**：未来将矩阵运算、系统级优化求解、蒙特卡洛模拟以及海量的本地 SQLite 模糊匹配检索统统用 Rust 重写。
    *   **💡 技术实现路径**：建立 Cargo workspace。采用 `napi-rs`，通过 `#[napi]` 宏暴露出 Rust 计算函数，在打包环节编译为不同操作系统的 `.node` 原生扩展模块，供 Electron 主进程直接、非阻塞地调用。对于渲染进程内的独立纯净计算，则通过 `wasm-pack` 编译为 `.wasm` 供前端 Worker 调用。
2.  **本地内存极限优化**：
    *   **💡 技术实现路径**：针对未来多达几十兆的大型非线性材料特性曲线数据，如果使用 Node.js 读取会导致严重的 JSON 反序列化延迟与 V8 GC (垃圾回收) 停顿。改为在 Rust 核心中使用内存映射文件（Memory-Mapped File, Mmap），Node.js 端仅通过 Buffer 指针直接访问所需的字节块，实现真正意义上的“零开销”极速读取。

## 7. 市场竞品对标与突围差异化策略

结合目前市面主流工业辅助软件的现状，MechBox 必须打出“降维打击”与“差异化体验”：

### 7.1 对标传统表格插件（如 MITCalc）
*   **市场现状**：MITCalc 功能极其全面，但重度依赖 Excel，UI 停留在上世纪 90 年代，交互极其繁琐，且极易出现单元格公式被误删的工程事故。
*   **MechBox 突围**：提供完全独立的现代客户端体验。计算底层黑盒化保护（Rust核心），但计算过程**白盒化展示**（界面上通过 LaTeX 优雅渲染公式推导过程，一键导出用于 ISO 9001 设计评审的标准 PDF 计算书）。

### 7.2 对标云端 SaaS 工具（如 MechaniCalc）
*   **市场现状**：界面现代，但高度依赖互联网，在军工、保密企业车间或无网现场完全瘫痪；采用高昂的按月订阅制，数据有泄露风险。
*   **MechBox 突围**：主打 **100% 本地离线安全**与**极速瞬时响应**。绝不让用户的一张图纸、一个参数流向外网，直击保密研发单位的核心痛点。

### 7.3 对标顶级巨头软件（如 KISSsoft、Romax）
*   **市场现状**：绝对权威，极其专业。但价格极为昂贵（动辄几十万授权费），学习曲线极其陡峭，对于日常小项目估算来说完全是“杀鸡用牛刀”。
*   **MechBox 突围**：主打**“工程师的快反特种兵”**。不盲目追求深度的有限元网格精度，而是提供基于国家/国际标准的**快速定性与边界估算**，以极简交互让工程师在 1 分钟内得出 80% 准确度的可行方案。

### 7.4 对标大厂自带工具（如 SKF、Parker 官网计算器）
*   **市场现状**：数据准，但存在严重的“厂商锁定”（Vendor Lock-in）。SKF 的工具绝对查不到 NSK 的数据，无法给采购提供多源参考。
*   **MechBox 突围**：做**“中立的工程聚合平台”**。在同一个选型界面下，横向拉齐 SKF、NSK、FAG、MISUMI 的同规格型号与参数进行平行对比，甚至允许导入“企业内定物料代码”，彻底打破大厂的信息壁垒。

---

## 结语：工业软件的终极壁垒
MechBox 的核心护城河不在于“连接云端”，而在于**“断网状态下的绝对可靠、资料的浩瀚与计算的全能”**。未来的 MechBox 将通过“多物理场/系统级耦合求解”、“超大规模本地工程百科知识库”、“与 CAD 的本机双向热链接”以及“参数化 3D 孪生反馈”，成为机械工程师在任何严苛或保密环境下最值得信赖的离线桌面指挥中心。
