# MechBox (MechCalc) 系统缺陷深度剖析与架构演进方案

> **文档定位**：本文档以极其苛刻的资深软件架构师及工业软件专家的视角，对当前 MechBox v1.0 的架构设计、计算引擎、数据管理及 UI/UX 进行深度“挑刺”，并提供面向工业级产品的演进解决方案。

---

## 1. 核心计算引擎：从“单向映射”到“约束求解”的致命缺陷

### 1.1 缺陷剖析：天真的纯函数映射
当前架构中，计算引擎被定义为纯函数（Pure Functions，如 `calcLife(C, P)`），本质上是 $Y = f(X)$ 的单向映射。
**痛点**：在真实的机械设计中，工程师往往面临的是**多变量约束求解（Constraint Solving）**或**目标寻优（Goal Seeking）**。
*   **场景**：我需要一个外径受限（$D \le 50$）、要求寿命 $L_{10} \ge 10000$ 小时的轴承，请给我推荐合适的型号和材料。
*   **现状**：目前的“纯函数”引擎只能做正向校验，无法处理这种带边界条件的逆向搜索和多目标优化。如果强行在前端用 `while` 循环遍历标准库，会导致严重的性能问题（UI 线程阻塞）。

### 1.2 解决方案：引入轻量级约束求解器
**架构重构**：
1.  **引擎升级**：在保持核心公式纯函数的基础上，在引擎上层封装一个**求解器层（Solver Layer）**。
2.  **算法引入**：对于简单的逆向计算，使用牛顿迭代法（Newton-Raphson）或二分法；对于复杂的选型（如弹簧刚度与尺寸的非线性关系），引入轻量级的约束满足问题（CSP）求解引擎或基于惩罚函数的启发式搜索。
3.  **Web Worker**：所有的迭代寻优计算**必须**放入 Web Worker 或 Electron 的单独 Node.js 子进程中执行，彻底解放渲染进程的 UI 线程。

## 2. 数据管理：静态 JSON 的“死亡陷阱”

### 2.1 缺陷剖析：硬编码与“伪离线”
将 GB/T、ISO 甚至材料的化学兼容性写死在 `data/**/*.json` 中，并在发版时打包。
**痛点**：
1.  **维护灾难**：工业标准会废止、更新。一旦发生（例如材料的许用应力在最新国标中被修正），唯一的办法是发一个新版本的安装包让全员更新。
2.  **团队孤岛**：某位工程师在本地新建了一个“非标” O 型圈规格或录入了一个特殊材料，同组的其他工程师完全无法复用，这在企业级应用中是不可接受的。
3.  **大内存开销**：当 JSON 膨胀到几十 MB 时（特别是包含复杂材料曲线的数据），在启动时全部反序列化加载到内存会极大地拖慢启动速度，增加 V8 引擎的 GC（垃圾回收）压力。

### 2.2 解决方案：本地 SQLite 搭配云端热更方案
**架构重构**：
1.  **废弃 JSON 静态全量加载**：全面拥抱 `better-sqlite3`。将所有标准库初始化导入 SQLite。按需查询，而不是启动时加载进内存字典。
2.  **云端热更新（OTA for Data）**：
    *   在服务器维护一个版本化的标准数据库。
    *   客户端在后台静默轮询版本号，发现更新后，下载增量的 SQL 补丁或新的 SQLite db 文件进行热替换（Hot Swap）。
3.  **本地增量扩展**：在 SQLite 中区分 `system_standards`（系统标准，只读）和 `user_standards`（用户自定义，可读写）。为企业版预留局域网同步接口。

## 3. 跨软件协同：无法打通 CAD 的“数据孤岛”

### 3.1 缺陷剖析：算完即止的尴尬
目前软件的终点是“导出 PDF 计算书”。
**痛点**：用户算出了齿轮的模数、齿数和变位系数，却还要打开 SolidWorks，凭着 PDF 的数据**手动重新输入**一遍来生成 3D 模型。这不仅低效，而且极易因“手抖”产生人为错误（Human Error）。缺乏与下游 CAD 软件的联动，是所有“计算器”类软件的天花板。

### 3.2 解决方案：构建宏/脚本导出桥梁
**架构重构**：
1.  **CAD 宏代码生成**：不仅导出 PDF，还可以导出 `.swp` (SolidWorks VBA 宏)、`.py` (FreeCAD/AutoCAD Python 脚本) 或通用的 `.step` 占位模型。
2.  **无缝衔接**：工程师在 MechBox 计算完齿轮或 O 型圈沟槽后，点击“导出到 SolidWorks”，MechBox 生成一段宏代码。用户在 CAD 中运行该宏，即可自动绘制出包含正确公差与沟槽特征的零件。这才是真正的“工业级提效”。

## 4. UI/UX 设计：过度互联网化，缺失“工程厚重感”

### 4.1 缺陷剖析：Ant Design 默认主题的“轻浮”
使用默认的 Ant Design Vue 5.0，大圆角、过多的留白、亮蓝色的主色调。
**痛点**：
1.  **信息密度极低**：在机械计算中，屏幕上同时需要展示输入参数、极限偏差表、装配简图、寿命曲线等。过大的留白会导致用户需要频繁滚动页面。
2.  **专业感不足**：默认的 Web 风格让它看起来像一个后台管理系统（Admin Dashboard），而不是一个严谨的工业辅助设计软件（如 Ansys, KissSoft, EPLAN 都是高信息密度、深色/冷灰基调）。

### 4.2 解决方案：打造高密度的“工业控制台”风格
**UI/UX 重构指南**：
1.  **布局紧凑化（Compact Mode）**：全面启用 Ant Design 的紧凑模式。减小表单的 `margin` 和 `padding`，字体大小调整为 `13px`，最大化同屏信息容量。
2.  **面板化与可拖拽（Docking & Panels）**：参考 Visual Studio Code 或工业 CAD 软件，摒弃单纯的“上中下”滚动网页布局。将“计算输入”、“标准库浏览”、“结果图表”拆分为可折叠、可拖拽的面板（Panels）。
3.  **色彩规范的克制**：
    *   主色调：舍弃亮蓝，改用更具工业感的“深邃蓝”（如 `#1E3A8A`）或“机械灰”。
    *   状态色严格语义化：红色只用于“校核不通过”或“危险警告”，绿色用于“校核通过”。不要在装饰性元素上滥用高饱和度色彩。
4.  **沉浸式双向联动图表**：左侧的输入参数修改时，右侧的装配简图（可以用 SVG 动态渲染）必须高亮显示正在修改的尺寸特征（例如输入沟槽深度时，剖面图的深度标注线变红并动态变化），做到所见即所得。

## 5. Electron 的原罪：性能与体积的权衡

### 5.1 缺陷剖析：重达 200MB+ 的“简易计算器”
**痛点**：Electron 打包的 Chromium 内核会让一个简单的计算工具占用极大的内存和磁盘空间，在车间工控机或老旧办公电脑上表现不佳。

### 5.2 解决方案：性能榨取与未来演进
1.  **Electron 极致优化**：
    *   在主进程中延迟加载非必要模块。
    *   关闭不必要的 Chromium 命令行开关（如关闭 GPU 硬件加速的某些特性，如果遇到渲染黑屏的低端机）。
    *   严格遵守 `contextIsolation`，使用 `invoke/handle` 进行 IPC 通信，避免序列化过大的对象。
2.  **长远架构备选**：如果未来应用形态稳定，且用户对体积极其敏感，下一代架构应考虑使用 **Tauri** (Go + Vue) 或 **Wails** (Rust + Vue)。它们复用操作系统的原生 Webview（Edge WebView2 / WebKitGTK），可将安装包体积从 100MB 缩减至 10MB 左右，内存占用骤降 70%。

---

## 结语：工业软件的护城河在于“严谨”与“生态”
MechBox 绝不能止步于一个“带 UI 的公式计算器”。必须通过**约束求解引擎**解决真实的反向选型痛点，通过**热更新数据库**保证标准的绝对权威，通过**高密度专业 UI**赢得工程师的信任，最终通过**输出 CAD 宏脚本**打通工程落地的最后一公里。这才是走向商业级工业软件的必由之路。
